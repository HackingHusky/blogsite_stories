Offsec Nibbles CTF Write-up

While studying for the OSCP, I came across a box called Nibbles from OffSec Proving Grounds. That became one of my favorite attack boxes after learning about the ordinary-day issues with PostgreSQL and my favorite developer quote of all time: "It's not a bug, it's a feature."

The box is relatively straightforward for the first part, but it becomes a bit more complicated later with privilege escalation. However, if you know what to look for, you can easily obtain the last flag.

The box is at a mid-level, but don't let that stop you. Part of the OSCP is, no matter how hard the box is, to enumerate, enumerate, enumerate. The OSCP model is to try harder, but honestly, if you remember Miss Fritzel's quote, "At first, as you don't succeed, ask why you failed the first time." This is likely a more effective mindset to have when taking the test. So in this box, we know it's a Linux OS.

As part of this lab, I like to export all my IP addresses to the target to simplify things for me. So for the sake of this how-to, I'll refer to the target's IP as $target:

By starting with the low-hanging fruit, you can access the FTP server on port 21, check the website on port 80, or, in this case, PostgreSQL is running on port 5437. Therefore, before gaining access to the other two ports, SQL can still be accessed or used for remote code execution. We can ignore ports 139 and 445 since both are currently closed.

Before I delve into the box, one thing that separates hobbyists and professional ethical hackers is understanding the frameworks and the CVSS. This is also important because during the OSCP, you need to conduct a report for the test; understanding the "why this is a security risk" will help you.

In the PostgreSQL vulnerability of almost any version, thanks to these CVEs: CVE-2019–9193, CVE-2022–1552, and CVE-2023–39417. Especially this one: CVE-2022–1552. The issue here, regardless of the PostgreSQL version, is that you can run a few scripts, especially in Python, to gain access to the server with just the username and password.

I focus on gaining root access before obtaining the flags, because by then, you have already compromised the machines. For the proving grounds, they are always in the user's home directory on Linux or in the Desktop on Windows.

Taking it a step further, though, for a real-life engagement, this box is easily compromised. Default passwords are a no-go. Even if this is in a sandbox, you should still use a stronger password than the default and restrict its access to the network. After post-exploitation, we can pivot off this box on the network using Chisel to see if we can gain access to DC. Additionally, with root access, we can simultaneously access both the passwd and shadow files and run John the Ripper with a password list to crack accounts.

Hardening PostgreSQL by implementing SSL encryption is crucial, as a basic Python script can access the host on the network. Additionally, applying Mac filtering firewall rules can help prevent rogue devices from accessing the system, which can be particularly beneficial in cases like my attack box. Above all, update the software.

The challenging aspect of this CVE or vulnerability is that it was deployed to production, labeled as a feature, and, for PostgreSQL, associated with a CVE; however, it cannot prevent an RCE. Even with human error, we need to remind ourselves to slow down and not rely solely on AI/Vibe coding, because what can be built for good can bite back tomorrow. "It's not a bug, it's a feature" can lead to numerous issues, as this "feature" is quite exploitable.

This was still one of my favorite boxes I came across while studying for the OSCP. Sometimes, dumb luck and simply googling can provide the answer you need.
