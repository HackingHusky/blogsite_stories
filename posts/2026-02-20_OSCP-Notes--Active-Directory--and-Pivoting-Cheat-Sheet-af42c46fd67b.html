OSCP Notes, Active Directory, and Pivoting Cheat Sheet

Active Directory. Not only the hardest thing to defend, but it is also the backbone of your business. It was designed for accessibility and interoperability back in the late 90's/early 2000’s. It’s the company’s phone book, data access, and user access. It’s the keys to the kingdom. It’s hard to defend against the inherent “Open by Default” architecture, the massive attack surface, and “Living off the Land” attacks. LDAP maps the network structure; built-in trust assumes everything inside the network is secure; insecure protocols like NTLM and LLMNR are easily abused for credential harvesting; misconfigured users can become “shadow admins”; Kerberoasting; misconfigured GPOs; DCShadow, etc. I could go on for days. However, this can be overwhelming for others starting.

Before the OSCP, I took the PJPT by TCM Security. To get a real feel for AD. That test wasn’t as bad as I thought, but for the OSCP, learning those concepts will help here. However, note that there is no LLMNR Posing. But you can still use responder. As long as you use this:

sudo responder -I tun0 -A -v

As long as you use -A, all posing features will be turned off. Or you can nano into it. I know you need to know how to use it to pass the PJPT. For this exam, it's an enumeration test so that you can use it, but you may not need it. Have it in your back pocket.

Now, what makes this test so hard is that it brings together multiple skills at once. You aren’t just exploiting one thing—you’re navigating an entire ecosystem: OU abuse, mismanaged users, etc.

Host → User → Group → Domain → Policies → Protocols → Trusts

In OSCP labs, the AD chain is difficult because you must:

Enumerate users
Enumerate permissions
Understand group memberships
Notice subtle privilege assignment
Identify misconfigurations across multiple machines

You have a set to do. You may or may not get a user's credentials. Depending on the situation, follow these rules:

Rule one: 
assume you have anonymous/guest access. Until you after you run nxc
and don't have a user, assume, that is a misconfigure, until otherwise. 

Rule two: 
if you have a user, test everything, record it in blood hound, 
enumerate that user. That is your very strong clue. 

Rule three: 
the chain follows Microsoft PC's, if you get a user, and get access
review windows privesc. You may need to gain local access first. 

Rule four:
LOW PRIV USER
      ↓
Find unusual privilege or misconfiguration
      ↓
Abuse that privilege to escalate to a higher-priv account
      ↓
Gain access to a more sensitive machine or service
      ↓
Enumerate again (permissions change!)
      ↓
Find next misconfiguration
      ↓
Repeat until Domain Admin

Rule five:
Ask, why do you have this? Re-enumerate and review notes on user.
“Who do I control?”  “What can that object do that I couldn’t do before?”
“Who does THAT affect?”


Rule six: 
Treat this like another HTB or Proving Grounds box. It's not real
world. Think of the different quick methods on enumeration. You may not even
need to use minikatz (I would still use it). 

Rule seven:
If a new account gives you:

A new permission
A new SPN
A new group membership
A new GPO
Access to a new machine

Rule eight:
Kerberos Misconfigurations

Misconfigured SPNs
Weak or unexpected delegation
Accounts allowed to impersonate others
Users with SPNs who shouldn’t have them

Rule nine:
OSCP isn’t trying to trick you — they want to see if you can filter 
signal from noise.

Which AD objects matter
Which permissions actually mean escalation
What can be ignored


Rule ten:
the exam is testing your ability to chain multiple small wins, not find one 
big vulnerability.

Rule elven:
Active Directory privesc:

“This user can reset that user’s password”
“This group has GenericWrite on that object”
“This machine account can add a new SPN”
“This GPO applies somewhere unexpected”

Rule twelve:
It's ok to get hashes, pass the hash is quicker, just test it first.

Use the attack patterns to your advantage:


Low-Priv User
    ↓
Find: User has GenericWrite on another user
    ↓
That user has SPN / special rights
    ↓
That account can manage a service or machine
    ↓
Machine permission leads to DC access
    ↓
Domain Admin


GPO risk:
Low-Priv User
    ↓
User can modify a GPO
    ↓
GPO applies to a server or privileged user
    ↓
Privileges escalate when that user logs in
    ↓
Server access eventually leads to DC
    ↓
Domain Admin

Machine account:
Low-Priv User
    ↓
Compromise a machine account
    ↓
Machine has unexpected rights on DC object
    ↓
Privilege escalates through ACL chain
    ↓
Domain Admin

Combined AD + Windows PrivEsc Logic (OSCP Winning Formula):

Gain initial foothold (web, SMB, RDP, WinRM).
Enumerate creds + AD data.
Try creds across protocols + machines.
Gain new user or machine account.
Repeat enumeration.
BloodHound analysis 
Identify misconfig path.
Move laterally.
Enumerate local Windows for PrivEsc.
Escalate to admin.
Dump new creds.
Repeat across domain.
Eventually escalate to Domain Admin (conceptual).

This process may not work for others, but stick to a process to help you be accountable. Make the checklist and take your time.

Welcome | NetExec
NetExec is 100% open source and built by developers just like you.www.netexec.wiki

Netexec and crackmapexec are your best tools for enumerating and validating accounts with services like LDAP, RDP, smb, WinRM, etc. I refer to this wiki page a lot. and the commands:

netexec smb $target -u  ‘’ -p ‘’ --shares
netexec smb $target -u ‘’ -p  ‘’ --rid-brute
netexec smb $target -u ‘’ -p ‘’ --pass-pol
nxc smb <target> -u <user> -p <pass> --local-auth
nxc smb <target> -u <user> -p <pass> --groups
nxc smb <target> -u <user> -H <hash> 
nxc smb <target> -u <user> -p <pass> --sam
nxc smb <target> -u <user> -p <pass> --lsa
nxc smb <target> -u <user> -p <pass> --ntds
crackmapexec smb <target> -u 'guest' -p '' --shares -> same as nxc

nxc ldap $ip -u $user -p $password --users
nxc ldap $ip -u $user -p $password --groups
nxc ldap $ip -u $user -p $password --query "(sAMAccountName=Administrator)" "sAMAccountName objectClass"
nxc ldap domain -u username -p password --gmsa
The LDAP protocol can be used in combination with options like --kerberoasting and --no-preauth-targets 

nxc rdp $target -u user -p password
--no-bruteforce

nxc winrm <target> -u <user> -p <password> -x "whoami"
nxc winrm <target> -u <user> -H <ntlm_hash>

nxc winrm <target> -u <user> -p <password> -X 'powershell -c "Get-Process"'

You want to enumerate as best you can before attacks, because a user may not have rights like smb but still have RDP access (a dumb example).

So huge tip: use what you like, know what will work on the OSCP exam, and do the labs to get a feel for pivoting. The concept, though, is the same:

You compromise a Windows or Linux machine
You run ipconfig or ifconfig and see multiple NICs or subnets
Example:
10.10.10.50/24   ← the subnet you already know about
172.16.5.20/24   ← NEW hidden internal subnet

You can't reach 172.16.5.0/24 directly from Kali…
…but the compromised machine can.

Identify that 10.1.50.0/24 is hidden
Set up a SOCKS proxy through the compromised host
Use proxychains to run(for chisel):
 
nmap
rpcclient
smbclient
ldapwhoami
bloodhound-python
