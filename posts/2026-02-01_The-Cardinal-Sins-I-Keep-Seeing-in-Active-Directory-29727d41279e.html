The Cardinal Sins I Keep Seeing in Active Directory

I've been working in IT/Cybersecurity for seven years now, and I keep finding common day-to-day issues that could be resolved with a simple change in Active Directory. Now I understand, onboarding new users, computers, and printers can be annoying. However, it doesn’t have to be as painful as people think. However, I keep seeing this all the time. Users are being created but aren't actively using security groups, and users aren’t going into Object Units (OUs) for short.

Now, if things get inherited, that’s one thing. However, I had to train a few System Admins/Help Desk staff to understand the repercussions of not using Security Groups when auditing a Domain Controller.

I created a Domain Controller as an example of what I saw:

Users are being created in the Forest but not being assigned to the correct OU. I seen that a lot. That’s a user, but it's set at the default. If this person is working in a department, they need ot use the correct OU. Like, does this person need HR? Is it a manager, etc.? But it's just directly there. Why? Then, checking the user itself:

No context, just a user, now checking their security groups or distribution groups.

So why is he a domain admin? So basically, you have a user you don’t know who they are or what they do, with RDP access, and they're a domain admin. But they have to keep that around because this account basically controls the business. This is all hypothetical in my Active Directory lab, but the point is that I've seen this in many companies. When it comes time to audit this person, they're compromised because they have to keep this account open and be a domain admin for xyz reasons. No, anyone who needs to access information must be added to a distribution group/security group. HR documents need to stay with them because they contain sensitive information, such as PII. IT shouldn’t really have access. Part of the reporting process is discovery. I have found a lot of this, and it needs to be fixed asap. You're basically giving out the keys to the castle, and wonder how you got robbed.

Ok, we have HR and a Security group, great. But also, this is the problem:

I also see this all the time. You have the security group, sure, but users can access the folder directly rather than through it, and you have a user with complete control over the share folder.

So, if this account is deleted, everyone who had full access to this folder will no longer have it, and recovering this shared folder is extremely painful because a user, rather than a group, now controls the parent folder. I see this all the time. A group or the Administrator should have control over this and remove users by default, because not everyone needs access to it. And if they do, you can make a child folder in the parent folder to restrict accounts. Threat actors will look for anything they can get their hands on, making it an easy target.

Again, though, this is all on my hypothetical Domain Controller I made as a lab, but I seen it a handful of times. So when you are onboarding a new user, don’t just think of this as a ticket to be completed, ask, “What is this user’s department, what needs to be restricted, and what level of access this person needs?” Ask for a user who has that level of access.

It’s a simple fix that may take an hour or two, but it can be hardened for Active Directory. Create new groups, add the users, sync with Azure Sync Server, and run gpupdate /force on the Domain Controller. Also, running an audit every quarter helps ensure users who are no longer with the company are disabled.

Also, running AccessEnum helps too:

That’s why Pentest is extremely important to check for these holes, it's to help find a hole and provide the necessary feedback to fix.
