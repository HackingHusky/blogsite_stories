LLMNR & NBT-NS Posing, My Favorite Attack!

I wanted to build out an Active Directory server for fun. One big topic that comes to mind in Pentesting is LLMNR & NBT-NS Posing.

Link-Local Multicast Name Resolution Protocol is a protocol that enables devices on a local network to resolve hostnames without relying on a DNS server. It uses both IPv4 and IPv6 to resolve host names on the same local network. It’s the successor to NetBIOS (Network Basic Input/Output System). It used an older protocol that heavily relied on an early version of Windows. So when you think of NBT-N, it's also a failback protocol when DNS resolution fails. It enables local name resolution within a Local Area Network (LAN). It’s basic. It's a failback: if the Domain Controller hosting Active Directory fails, DNS and everyone on the network can still access resources, such as shared folders via SMB on ports 135 or 445.

Yeah, that’s a lot of knowledge to throw at a beginner. However, it’s still today, and it's my tutor's attack. Now, if you're studying for the OSCP, you will fail if you try this attack. However, this is an attack that a Pentester/Ethical Hacker should know. This is a widespread attack on Active Directory. And the reason is the OSCP ban? You get credentials without really trying.

So, step one of this attack: someone is trying to access a folder path over SMB (445). So let's access the \\hackme shared folder on the server. The DNS server doesn’t really know, but an attacking PC is broadcasting its IP address and attacking its own file share server. Then someone tries to access it, boom. You get the hashes of users attempting to access smb:

On my own home lab, I was able to get the hashes using responder, and here's how. Now, there is a lot of knowledge you get thrown about this attack; however, running it is extremely easy:

sudo responder -I eth0 -dw

The responder can already perform the attack. When conducting a Pentest, it's best to do so in the morning, when everyone is logged in to their machines. Once you have the user hashes, you can crack them offline with either Hashcat or John the Ripper.

john <hash.txt> --wordlist=<wordlist.txt>

hashcat -m 5600 -a 0 <hash.txt> <wordlist.txt>

So after getting this cracked (this is a bad example since that’s the user domain admin access, you practically won then). Then you can start with the following steps: running Bloodhound, Netexec on smb, RDP, LDAP, Impacket-GetSPNs, WinRM, etc. You want to check the user's access. Then use Mimikatz, Impact-PSEXC, and xfreerdp to access the Domain, or even run smb relay attacks. Possibilities open up.

So why is it so effective? It exploits default and legacy behaviors in the Windows network. It's meant for convenience rather than security (Note: you can use responder on the oscp, but you have to disable LLMNR posing). It intercepts credentials, checks for horizontal/vertical movement, and escalates privileges. Honestly, it's my go-to on a pentest, and it's extremely common.

So how does one defend it? The question really depends on the company. If they need it open, they need to implement Network Access Controls and stronger, more complex passwords. However, that’s still one easy way around the issue, so does the company want to accept that risk? Now, if they don’t, then the answer lies entirely with the Domain Controller. That way, Active Directory can be tightened up.

Step One:

Coffee, always get coffee.

Step two:

Go to Group Policy Manager, drop down the forest to the Domain, right-click Default Domain, then click edit:

In Group Policy Management Editor:

Computer Configuration > Administrative Templates > Network > DNS Client. Click on “Turn off multicast name resolution”:

Open Policy, click Enabled, and next Apply:

Next, open PowerShell or cmd. Then run gpupdate:

gpupdate /force

We want to force the Policy as soon as possible. NoPolicy step three.

Step three:

Go to the control panel.

Control Panel > Network and Internet > Network Connections > TCP/IPv4 > Advanced tab > WINS tab, and then select “Disable NetBIOS over TCP/IP” in the Active Directory tab (Note: this only works locally):

Now we can add a PowerShell startup script to disable NBT-NS via a GPO:

set-ItemProperty -Path HKLM:\SYSTEM\CurrentControlSet\services\NetBT\Parameters\Interfaces\tcpip* -Name NetbiosOptions -Value 2

Back to Group Policy Management Editor:

Computer Configuration > Policies > Windows Settings > Scripts > Startup:

Hit Add, then open Notepad to add a command. Save it as disableNBTNS.ps1. I usually save the script to a new folder or a script folder I create; for practice, saving it to the Desktop folder is OK.

Next, click Browse and find the script, then click Open. Click Apply, then OK. Now, when the server starts up, this will block NBT-NS:

Step four:

Step four is our last step, and that is to run these in PowerShell:

$(Get-ItemProperty -Path "HKLM:\Software\Policies\Microsoft\Windows NT\DNSClient" -name EnableMulticast).EnableMulticast
wmic nicconfig get caption,index,TcpipNetbiosOptions

Just confirming our work:

And that should do it, if you genuinely don’t need that service. Also, you want to do this with mirror Domain Controllers. If your company has only one DC, you need to mirror another, at least in case one goes out, even though this attack is like a rogue Domain Controller itself.

So the question is: do you mitigate by needing this service, or not? Not all infrastructure is the same. It depends on the service, the company's size, and the extent to which it is relied on. The question is: do you turn off the service, or is it now essential and can’t be disabled? That becomes situational.

Part of the reason I love Active Directory attacks is that they depend on the situation and are really hard to defend against. But there are still places you can protect, too. But there is one final suggestion I love more than anything for defending Active Directory: a Honey User.

Like a Honey Pot or Honey Net, a Honey User is a fake account that a SIEM tool can detect via Event ID 4648. By combining that fake folder on a file server with the fake account, you can trap more flies. Yes, you would need more rules for your SIEM, but I also suggest implementing this to stop attackers during the reconnaissance stage of credential harvesting. That’s also an excellent option for smaller companies.
