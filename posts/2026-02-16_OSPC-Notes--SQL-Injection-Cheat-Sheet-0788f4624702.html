OSPC Notes, SQL Injection Cheat Sheet

Now, this is a huge topic to cover beyond the common attack types and kinda deserves its own section. However, SQLMap is not allowed on the test, so there will be no notes on that. OSCP wants 100% manually enumeration; however, does that mean you can’t automate an SQL attack with Python? You sure can! You can’t use recon tools to gain shells; this is a methodology test.

So what is SQL Injection? It’s like a naughty kid who figures out that if they shout a tricky sentence, they can break the lock on a toy box. Instead of saying “Open Sesame,” they shout, “Open Sesame OR let me in!”

So how does this work?

[1] User Input
    └─> login form fields:
         - username
         - password

[2] Insecure App Code (vulnerable sink)
    └─> Builds SQL by string concatenation:
         "SELECT * FROM users
          WHERE username = '" + username + "'
            AND password = '" + password + "'"

[3] Database
    └─> Executes the resulting SQL as-is

[4] Outcomes
    a) Normal case:
       - Safe input → intended query → normal auth flow.
    b) Injection case:
       - Input contains SQL syntax that changes query logic
         (e.g., bypass WHERE clause or add UNION)
       - DB returns unintended rows / errors / metadata

Union base attacks are different. a technique where an attacker abuses the SQL UNION operator to combine the results of the original query with additional, attacker‑controlled queries. How the union base works:

[User Input]
   ↓
Application (vulnerable)
   ↓
SQL Query Sent to DB:
   SELECT id, title FROM posts WHERE id = <input>;

Attacker Manipulates Input:
   <input>  →  closes query + adds UNION clause

DB Executes:
   SELECT id, title FROM posts WHERE id = 1
   UNION
   SELECT id, email FROM users;
